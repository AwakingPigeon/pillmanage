# 项目管理规则文档

## 1. 对话记录规则

### 1.1 对话记录要求
- **文件名**：所有对话记录必须保存在 `chatlog.md` 文件中
- **记录频率**：每次重要的对话交流后都需要更新对话记录
- **记录内容**：包括对话时间、主要内容、关键决策点、文档变更等
- **格式标准**：使用Markdown格式，保持结构清晰

### 1.2 对话记录结构
```markdown
# 项目对话记录

## [日期] 对话记录

### 对话主题
[简要描述本次对话的主题]

### 主要内容
- 用户需求/问题描述
- 助手分析/建议
- 关键决策点
- 文档/代码变更

### 结果产出
- 创建/修改的文件列表
- 重要决策记录
- 下一步行动计划
```

## 2. 文档管理规则

### 2.1 文档命名规范
- 主要文档使用英文命名
- 辅助文档可以使用中文命名
- 文件名应简洁明了，反映文档内容

### 2.2 文档更新流程
1. 用户提出需求或问题
2. 助手分析并提供解决方案
3. 实施变更（修改文档/代码）
4. 更新对话记录
5. 确认变更结果

## 3. 项目文件结构规范

### 3.1 必需文件
- `PRD.md` - 产品需求文档（主文档）
- `chatlog.md` - 对话记录文档
- `project_rules.md` - 项目管理规则文档

### 3.2 可选文件
- 根据项目需要创建其他技术文档
- 代码文件按功能模块组织

### 3.3 远程仓库配置
- **远程仓库地址**：`https://github.com/AwakingPigeon/pillmanage.git`
- **分支管理**：主分支为`main`，功能开发使用特性分支
- **提交规范**：提交信息需清晰描述变更内容
- **同步频率**：重要变更后及时推送到远程仓库

## 4. 变更管理规则

### 4.1 文档变更
- 所有文档变更需要在对话记录中说明
- 重要变更需要记录变更前后的对比
- 变更原因和影响需要明确记录

### 4.2 代码变更
- 代码变更需要说明修改目的
- 复杂变更需要记录技术方案
- 测试结果需要记录在案

## 5. 沟通协作规则

### 5.1 需求沟通
- 用户需求需要明确具体
- 助手需要确认理解正确
- 复杂需求需要分解为可执行步骤

### 5.2 进度跟踪
- 使用todo列表跟踪任务进度
- 定期更新项目状态
- 记录已完成和待完成的工作

## 6. 公益项目原则规则

### 6.1 项目定位原则
- **纯公益性质**：本项目为完全公益性质，不涉及任何商业盈利
- **用户免费使用**：所有功能对用户完全免费开放
- **无商业合作**：不引入任何商业机构合作（药店、保险公司等）
- **无数据商业化**：用户数据仅用于改善产品体验，绝不用于商业用途

### 6.2 功能边界原则
- **专注核心功能**：仅提供基础的服药提醒和记录功能
- **避免医疗建议**：不提供任何医疗建议或诊断功能
- **本地数据存储**：所有数据仅存储在用户设备本地
- **无第三方集成**：不集成任何第三方服务或API

### 6.3 运营透明原则
- **开源透明**：项目代码开源，运营过程透明
- **社区驱动**：功能开发基于用户真实需求
- **持续维护**：确保产品的长期可用性和稳定性

## 7. 质量保证规则

### 7.1 文档质量
- 文档内容准确无误
- 格式规范统一
- 逻辑清晰连贯

### 7.2 代码质量
- 代码符合编程规范
- 功能实现完整
- 必要的错误处理

---

## 规则生效

本规则文档自创建之日起生效，所有项目参与者都需要遵守上述规则。规则内容可以根据项目需要进行调整和补充。

## 特别声明

本项目为纯公益性质，所有开发工作都基于帮助抑郁症患者解决服药健忘问题的公益目的。任何功能开发都必须符合公益项目原则，不得引入商业元素。
